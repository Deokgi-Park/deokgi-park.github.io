# 한국인용 Jekyll(Minimal Mistakes) 블로그 운영 지침

이 문서는 Minimal Mistakes 공식 Quick Start를 바탕으로, 이 저장소(`deokgi-park.github.io`)를 한국어 환경에서 안정적으로 운영하기 위한 실무 지침입니다.

참고 원문:

- https://mmistakes.github.io/minimal-mistakes/docs/quick-start-guide/

## 1. 이 저장소의 현재 운영 방식

현재 설정을 기준으로 보면:

- `Gemfile`은 `github-pages` + `jekyll-include-cache` 조합 사용
- `_config.yml`의 `theme:`/`remote_theme:`는 주석 처리
- 저장소에 `_includes`, `_layouts`, `_sass` 등이 직접 포함

즉, 현재는 **Minimal Mistakes 파일을 저장소에 직접 포함한 방식(포크/복사형)**에 가깝습니다.

## 2. 최초 1회 로컬 환경 준비(Windows)

아래 문서를 먼저 수행하세요.

- `doc/local에서 구동`

핵심 명령:

```bash
bundle install
bundle exec jekyll serve
```

접속 주소:

- `http://127.0.0.1:4000`

## 3. 일상 운영 루틴(가장 많이 쓰는 흐름)

### 3-1. 새 글 작성

파일 위치:

- `_posts/YYYY-MM-DD-파일명.md`

권장 Front Matter 예시:

```yaml
---
title: "글 제목"
date: 2026-03-09 09:00:00 +0900
categories: [dev, jekyll]
tags: [jekyll, minimal-mistakes]
layout: single
author_profile: true
toc: true
---
```

작성 팁:

- 이미지 경로는 가급적 절대 경로 사용: `/assets/images/파일명.png`
- 한글 파일명보다 영문 slug(파일명) 사용 권장(링크 안정성)

### 3-2. 로컬 미리보기

```bash
bundle exec jekyll serve
```

변경 후 확인 포인트:

- 홈 목록 노출 여부
- 글 본문 스타일/코드블록/TOC
- 모바일 레이아웃 깨짐 여부

### 3-3. 배포

GitHub Pages 사용자 페이지 저장소(`USERNAME.github.io`)는 보통 `main` 푸시로 자동 배포됩니다.

```bash
git add .
git commit -m "Add new post"
git push origin main
```

배포 확인:

- `https://deokgi-park.github.io`
- 반영 지연 시 1~5분 대기 후 새로고침

## 4. Minimal Mistakes Quick Start 기준 필수 체크

### 4-1. `jekyll-include-cache` 유지

공식 문서에서도 강조하는 필수 항목입니다.

- `Gemfile`에 `jekyll-include-cache` 포함
- `_config.yml`의 `plugins` 배열에 `jekyll-include-cache` 포함

누락 시 대표 오류:

- `Unknown tag 'include_cached'`

### 4-2. 한국어 UI

이미 `locale: "ko-KR"` 설정이 있으므로 유지하세요.

추가 커스터마이즈 파일:

- `_data/ui-text.yml` (버튼/레이블 문구)
- `_data/navigation.yml` (상단 메뉴)

### 4-3. 이미지 경로 규칙

Quick Start의 v4 관련 변경사항에 맞춰, 이미지 참조는 전체 경로를 권장합니다.

예:

- `image: /assets/images/sample.jpg`

## 5. 테마 업데이트 전략(중요)

현재 저장소는 테마 파일을 직접 포함한 구조이므로, 일반적인 `remote_theme` 방식처럼 간단히 버전만 올리기 어렵습니다.

안전한 업데이트 절차:

1. 현재 동작 상태 태그/백업 브랜치 생성
2. Minimal Mistakes 변경 내역(릴리스/Changelog) 확인
3. 필요한 파일만 선별 반영
4. 로컬 빌드 + 핵심 페이지 점검 후 머지

주의:

- `_includes`, `_layouts`, `_sass` 직접 수정분이 많을수록 업데이트 충돌 위험이 커집니다.

## 6. 자주 발생하는 오류와 해결

- `bundle: command not found`
  - 터미널 재시작
  - Ruby 경로(PATH) 확인
- native extension 빌드 실패(예: `ffi`)
  - `ridk install` 재실행 후 MSYS2/GCC 설치 확인
- `Unknown tag 'include_cached'`
  - `jekyll-include-cache` gem/plugins 설정 재확인
- 로컬은 되는데 GitHub Pages에서 깨짐
  - 플러그인 사용 가능 범위(`github-pages` 호환) 확인
  - `baseurl`, 정적 리소스 경로(`/assets/...`) 점검

## 7. 운영 권장 규칙

- 큰 변경 전에는 브랜치 분리
- 게시글 작성/수정 커밋과 테마 수정 커밋 분리
- `_config.yml` 변경 시 서버 재시작 후 검증
- 배포 직후 데스크톱/모바일 둘 다 확인

## 8. 콘텐츠 종류별 사용법 (`post`, `page`, `portfolio`, `collection`)

이 저장소에서 주로 쓰는 콘텐츠 타입은 아래 4가지입니다.

| 종류       | 위치          | URL 규칙                | 추천 용도                               | 핵심 포인트                                               |
| ---------- | ------------- | ----------------------- | --------------------------------------- | --------------------------------------------------------- |
| post       | `_posts/`     | `/:categories/:title/`  | 블로그 글, 기술 글, 회고                | 파일명 `YYYY-MM-DD-slug.md` 필수                          |
| collection | `_<name>/`    | `/:collection/:path/`   | 날짜와 무관한 구조화 콘텐츠             | `_config.yml`의 `collections`에 먼저 정의 필요            |
| portfolio  | `_portfolio/` | `/:collection/:path/`   | 프로젝트/작업물 상세 페이지             | `_config.yml`의 `collections.portfolio.output: true` 기반 |
| page       | `_pages/`     | `permalink`로 직접 지정 | 소개, 연락처, 아카이브 같은 고정 페이지 | 날짜 없이 독립 페이지로 운영                              |

참고:

- `_config.yml` `defaults`에 의해 `posts`는 `layout: single`, `author_profile: true`가 기본 적용됩니다.
- `_config.yml` `defaults`에 의해 `portfolio`는 `layout: single`, `author_profile: false`가 기본 적용됩니다.

### 8-1. `post` 사용법

파일명 규칙:

- `_posts/2026-03-09-jekyll-start.md`

권장 템플릿:

```yaml
---
title: "글 제목"
date: 2026-03-09 09:00:00 +0900
categories: [dev, jekyll]
tags: [minimal-mistakes, guide]
layout: single
author_profile: true
toc: true
toc_label: "목차"
toc_icon: "list"
---
요약 한두 문장.
```

사용 팁:

- `categories`는 URL에도 반영되므로 너무 자주 바꾸지 않는 것이 좋습니다.
- 오타 주의: `auther_profile`이 아니라 `author_profile`입니다.

### 8-2. `portfolio` 사용법

`portfolio`는 컬렉션 기반입니다. 보통 아래 2단으로 운영합니다.

- 목록 페이지: `_pages/portfolio.md` 또는 `_portfolio/portfolio.md`에서 `layout: collection`
- 개별 항목: `_portfolio/*.md`에서 `layout: single`

개별 항목 권장 템플릿:

```yaml
---
title: "프로젝트 이름"
excerpt: "프로젝트 한 줄 소개"
header:
  image: /assets/images/project-cover.jpg
  teaser: /assets/images/project-cover.jpg
categories: [portfolio]
tags: [react, nodejs]
layout: single
permalink: /portfolio/project-slug/
---
프로젝트 개요
## 문제
## 해결
## 결과
```

목록 페이지 템플릿(`layout: collection`):

```yaml
---
title: "Portfolio"
layout: collection
permalink: /portfolio/
collection: portfolio
entries_layout: grid
classes: wide
---
```

### 8-3. `collection` 일반 사용법

`collection`은 `post`처럼 날짜 중심이 아니라, 독립 문서를 구조적으로 모으는 기능입니다.

예: `docs`, `notes`, `books`, `recipes`, `projects`

1. `_config.yml`에 컬렉션 정의:

```yaml
collections:
  notes:
    output: true
    permalink: /:collection/:path/
```

2. 폴더 생성:

- `_notes/`

3. 컬렉션 아이템 작성 템플릿:

```yaml
---
title: "문서 제목"
layout: single
permalink: /notes/note-slug/
author_profile: false
---
문서 내용
```

핵심:

- 컬렉션명 `notes`라면 폴더는 반드시 `_notes`
- `_config.yml` 정의와 폴더명이 다르면 빌드에 포함되지 않음
- `portfolio`도 결국 `collection`의 한 종류

### 8-4. `page` 사용법

`page`는 소개/문의/아카이브 같은 고정 페이지에 적합합니다.

기본 템플릿:

```yaml
---
layout: single
title: "About"
permalink: /about/
author_profile: false
---
페이지 본문 내용
```

목록형 페이지 템플릿(글 리스트를 보여줄 때):

```yaml
---
layout: posts
title: "소개 글"
permalink: /intro/
---
```

### 8-5. 언제 무엇을 쓰면 좋은가

- 연속적으로 발행되는 글: `post`
- 날짜와 무관한 독립 문서 묶음: `collection`
- 프로젝트/작업물 케이스 스터디: `portfolio`
- 사이트 안내/고정 문서: `page`

### 8-6. 실제 샘플(이 저장소에 추가됨)

요청하신 대로 `sample` 컬렉션 샘플을 추가해둔 상태입니다.

- 컬렉션 정의: `_config.yml`의 `collections.sample`
- 샘플 아이템: `_sample/sample-item.md`
- 샘플 목록 페이지: `_pages/sample-collection.md`
- 확인 URL:
  - `/sample/`
  - `/sample/sample-item/`

## 9. 실무 체크리스트

- 새 콘텐츠 생성 시 Front Matter 오타 먼저 점검
- 이미지 경로는 `/assets/images/...` 절대 경로 사용
- `permalink` 충돌(동일 URL) 없는지 확인
- 로컬 서버에서 모바일 화면까지 확인 후 푸시

## 10. VS Code 스니펫 사용법(처음 사용자용)

이 저장소에는 아래 스니펫 파일이 추가되어 있습니다.

- `.vscode/jekyll.code-snippets`

### 10-1. 기본 사용 순서

1. 새 Markdown 파일을 엽니다. (`_posts/*.md`, `_portfolio/*.md`, `_pages/*.md`)
2. 에디터에서 스니펫 prefix를 입력합니다.
3. 추천 목록에서 스니펫을 선택하고 `Enter`를 누릅니다.
4. `Tab` 키로 다음 입력 칸으로 이동하면서 값을 채웁니다.
5. 마지막 칸까지 입력 후 저장합니다.

### 10-2. 사용할 prefix 목록

- `jpost`: 일반 블로그 글 템플릿 (`_posts`용)
- `jcollection`: 일반 컬렉션 아이템 템플릿 (`_<name>`용)
- `jportfolio`: 포트폴리오 상세 템플릿 (`_portfolio`용)
- `jpage`: 고정 페이지 템플릿 (`_pages`용)
- `jposts-page`: 글 목록 페이지 템플릿 (`layout: posts`)
- `jcollection-page`: 일반 컬렉션 목록 페이지 템플릿 (`layout: collection`)
- `jportfolio-page`: 포트폴리오 목록 페이지 템플릿 (`layout: collection`)

### 10-3. 실제 예시

예: `_posts/2026-03-09-jekyll-snippet.md` 파일에서 `jpost` 입력 후 `Enter`

- 제목, 카테고리, 태그, 소제목 칸이 자동 생성
- `Tab`으로 항목 이동하며 빠르게 작성 가능

### 10-4. 자주 막히는 지점

- 스니펫 목록이 안 뜨는 경우:
  - 파일 확장자가 `.md`인지 확인
  - 입력 중 `Ctrl+Space`로 자동완성 목록 강제 호출
- 날짜/slug 규칙 누락:
  - `post`는 파일명 `YYYY-MM-DD-slug.md` 규칙 유지
- URL 충돌:
  - `permalink`가 기존 페이지와 겹치지 않는지 확인
